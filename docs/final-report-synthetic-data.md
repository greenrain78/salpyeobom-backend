## 1. 시스템 개요 (4-1)

### 1.1 이 서브시스템은 왜 필요한가

살펴봄은 독거노인의 생활(ADL: Activities of Daily Living) 센서 데이터를 받아 **평시·응급·사망**
세 상태를 분류하는 조기 모니터링 백엔드다. 그런데 분류기를 학습시키려는데 **검증된 실데이터가
`adl_raw_records` 의 id 1~60, 단 60행뿐**이었다 — 응급 대상자 1명(30일) + 사망 대상자 1명(30일).
**평시 클래스는 라벨된 실데이터가 아예 없다.**

즉 "1명 × 2클래스, 60일 표본"에서 "1000명 × 3클래스"를 일반화해야 하는, 데이터가 절대적으로
부족한 상황이었다. 이 서브시스템의 목적은 **그 60행의 분포·클래스 시그니처를 충실히 재현하는
합성 학습데이터(클래스별 1000명 × 60일 = 180,000행)를 생성**하는 것이다.

### 1.2 누구를 위한, 어떤 가치인가

- **대상**: ADL 분류기를 학습시켜야 하는 우리 팀(및 후속 모델 개발자).
- **가치**: 추가 실데이터 수집 없이 분류기를 학습·검증할 데이터 기반을 만든다. 단순 통계
  시뮬레이터 대신 **"환자 1명의 60일 인생을 쓰는 작가"** 방식(시나리오 기반)을 택해,
  (a) 서사가 모든 지표를 한 방향으로 묶는 **인과 일관성**, (b) 1000명이 서로 다른 삶을 갖는
  **다양성**, (c) 각 행에 "이유(스토리)"가 붙어 추적 가능한 **디버깅성**을 동시에 얻는다.

### 1.3 포함 범위 / 의도적 제외 (Scope / Non-Scope)

| 구분 | 내용 | 왜 |
|------|------|----|
| **포함** | LLM이 persona·scenario·60일 일과표(daily) 생성 | 서사·행동 단서는 창의·인과의 영역 |
| **포함** | 결정론적 익스팬더가 일과표 → 1440 분단위 배열 + 스칼라 전개 | 대량·일관·검증 가능한 정량화는 코드의 영역 |
| **포함** | 배치 파이프라인(JSONL/DB 싱크) + 실 vs 합성 분포 검증(KS) | 재현성·합격기준 확인 |
| **제외** | **LLM이 1440 배열을 토큰으로 직접 출력** | 180k행 × 행당 ~5,856숫자 ≈ **10억 토큰**. 비용·일관성·분포 모두 비현실적 → 코드가 전개 |
| **제외** | **id 1~60 외의 데이터·외부 통계를 근거로 사용** | "분포는 id 1~60 에서만 도출" 원칙. 외삽 범위를 명시적으로 통제 |
| **제외** | **분류기 학습 자체** | 사용자가 명시적으로 이번 범위에서 제외("분류기 학습을 제외한 나머지 모두 수행") |

> **목적–기능 연결 요약:** 목적은 *"60행 실데이터의 분포·클래스 시그니처를 재현한 학습데이터를
> 만든다"* 이고, 이를 위해 *시나리오 기반 LLM 생성 + 결정론적 익스팬더 + KS 검증*을 넣고,
> *LLM의 직접 배열 출력*과 *id 1~60 밖 데이터*는 비용·통제 이유로 의도적으로 뺐다.

## 2. AI 도구를 구체적으로 어떻게 썼는가 (4-2)

**도구:** Claude Code (Agentic Coding CLI). **DB:** 원격 PostgreSQL(Tortoise ORM).

### 2.1 단계별 사용

| 단계 | AI 활용 | 사람이 한 일 |
|------|---------|--------------|
| 목표 정의 | `/goal` 슬래시 커맨드로 생성기 사양(역할·입력·출력 JSON 계약·불변값)을 고정 | 사양 자체를 사람이 설계·작성 |
| 데이터 이해 | 원격 DB에서 id 1~60 프로파일링(분포·이산레벨·제로비율) 스크립트 작성·실행 | 어떤 지표를 봐야 하는지 지정, 결과 해석 |
| 설계 | 시나리오→일과표→익스팬더 파이프라인 구현 | 아키텍처 승인, 데이터 충실성 우선 원칙 강제 |
| 구현 | `scenario_gen.py`·`expander.py`·`run_batch.py`·`validate_batch.py` 작성 | 단계별 검토·게이트 승인 |
| 검증 | 2-표본 KS 직접 구현, 실 vs 합성 비교 | KS 결과의 타당성 판단, 과적합 경계 결정 |

### 2.2 실제 입력 프롬프트 인용과 분석

**프롬프트 A — 데이터 조회 (좋은 프롬프트)**

```
DB에 접속해서 adl_raw_records 의 id 1~60 데이터 조회 로컬 DB가 아닌 원격 DB에 접속
```

분석: 짧지만 **대상 테이블·정확한 ID 범위·로컬/원격 구분**을 모두 지정했다. 특히 "로컬이 아닌
원격"이라는 한정이 결정적이었는데, 이 프로젝트는 `.env`(원격)와 `.env.local`(로컬)을 병합 로드하고
**로컬이 우선**하도록 설정돼 있어, 이 한정이 없었다면 AI가 기본 설정대로 로컬 DB(빈 테이블)를
조회했을 것이다. → *좋은 프롬프트는 "무엇을"뿐 아니라 흔히 틀리는 축(여기선 접속 대상)을 미리
못 박는다.*

**프롬프트 B — 생성기 사양 `/goal` (강한 프롬프트, 부록 A에 전문)**

핵심 발췌:

```
## 출력 (JSON)
{ "class": "평시"|"응급"|"사망", "persona": {...}, "scenario": "...",
  "daily": [ {day, sleep, activity_by_hour[24], bath_events, outgoing_windows, env_base} ×60 ],
  "records": {...} }

## 지켜야 할 불변값 (후속 코드가 강제 — 일과표도 여기에 맞춰라)
- 사망 클래스는 목욕이 0 이다 → bath_events 는 항상 [].
- aix 비제로 값은 실데이터에서 관측된 이산 레벨 집합에서만 나온다 (… 약 71%는 0).
- outgoing 의 254·255 는 외출이 아니라 센서 무신호(sentinel) 다.

## 근거 데이터
모든 정량 분포·패턴은 adl_raw_records 의 id 1~60 에서만 도출한다.
```

분석: 이것이 **"기능 만들어 줘"와 좋은 프롬프트의 차이**를 보여준다. ① **출력 계약(JSON 스키마)**을
못 박아 자유서술을 차단했고, ② **불변값**으로 클래스 경계를 코드가 강제하도록 책임을 분리했고,
③ **근거 데이터 범위(id 1~60에서만)**로 환각·외삽을 통제했고, ④ **다양성 요구**로 반복을 막았다.
즉 프롬프트가 *창의의 영역(서사)*과 *강제의 영역(시그니처)*을 명확히 갈라, LLM이 "사망인데 목욕
5회" 같은 분류기 파괴적 변주를 만들지 못하게 설계됐다.

### 2.4 검토/승인 게이트와 그 활용

이 작업에서 **하네스가 제공한 게이트가 실제로 작동**했다(증거는 4-3에 상세):

## 3. 사람의 판단이 들어간 지점 (Human-in-the-Loop, 4-3)

> 이 장이 보고서의 핵심이다. AI가 만든 것을 **어떻게 검증·거절·수정·통제했는가**.

### 3.1 LLM 제안을 거절/수정한 사례

**사례 ① — 서사보다 데이터를 택함 (사망자 외출 곡선)**

LLM은 사망 시나리오를 "외출이 **D-20 전후로 끊긴다**"로 그럴듯하게 썼다(고립 심화 서사).
그러나 실데이터를 확인하니 사망 대상자의 `outgoing_count_d` 는 30일 내내 **평균 5**였다.
검증에서 합성값(평균 2)이 실측과 크게 어긋났다(KS 기각). 사람은 **"정량값은 id 1~60 분포를
따른다"는 원칙을 서사보다 우선**시켜, 익스팬더의 감쇠식과 시나리오 문구를 모두 수정했다.

```python
# 변경 전 (서사 주도): 외출이 D-20 전후 0으로 급감
out_n = max(0, round(2 * (1.0 - 1.6 * p)))
# 변경 후 (데이터 주도): 말기까지 완전히 끊기지 않음 — 실측 avg 5 재현
out_n = max(0, round(3 * (1.0 - 0.7 * p)))
```

→ *AI가 만든 "그럴듯한 이야기"를 실측 분포라는 외부 근거로 반증하고 교정한 사례.*

**사례 ② — `settings.DATABASE_URL` 함정을 사전 차단**

원격 DB 적재 코드를 짤 때 LLM의 자연스러운 선택은 `from app.config import settings;
settings.DATABASE_URL` 이었다. 그러나 사람이 **이 설정이 `.env`→`.env.local`을 병합하고
로컬이 우선**임을 상기해, 그대로 뒀다면 **원격이 아니라 로컬 DB로 조용히 적재**됐을 것임을 잡아냈다.

```python
# 함정: settings 는 .env.local(로컬)이 우선되어 원격 적재가 로컬로 새어나감
# 교정: 원격 적재 시 .env 의 DATABASE_URL 을 직접 읽음(읽기 전용)
db_url = args.db_url or _env_database_url()
```

→ *"동작은 하지만 조용히 틀린" 코드. 실행 결과만 봐선 안 잡히고, 시스템 설정을 이해해야 잡힌다.*

**사례 ③ — KS 결과를 교차검증해 AI의 측정 버그를 잡음**

AI가 처음 구현한 2-표본 KS는 **사망 `bath_count_d` 를 "D=1.00, p=0(완전 불일치)"**로 보고했다.
그런데 같은 화면의 클래스 분리도 표는 실·합성 **모두 목욕 0(완전 일치)**라고 말하고 있었다.
이 **모순**을 근거로 KS 구현을 의심했고, p-value 급수가 D=0에서 진동하며 0으로 수렴하는
수치 버그(동점 처리)임을 찾아 Numerical Recipes `probks` 로 교체했다(이후 p=1.000으로 정상화).

→ *AI 산출 숫자를 독립적 신호(분리도 표)와 대조해 "도구 자체의 결함"을 잡은 사례. 단일 숫자를
그대로 믿지 않았다.*

### 3.2 LLM이 잘못된 가정을 사실처럼 처리한 사례

- **night_aix "대부분 0" 가정의 오류**: AI는 처음 night_aix를 "초반 0, 종반 급등"으로 모델링했다.
  그러나 실데이터 정렬값을 직접 조회하니 **0은 4개뿐, 66~20417에 광범위 연속 분포**였다. 추측이
  아니라 **경험분포를 조회**해야 진실이 드러났고, 샘플러를 밴드 분포로 다시 짰다(KS p 0.000→0.537).
- **전수 KS의 과잉 기각 함정**: n=1500 전수 KS로는 다수 지표가 p<0.05로 기각됐다. 이를 곧이곧대로
  믿으면 "합성데이터가 나쁘다"고 잘못 결론지었을 것이다. 사람은 **실데이터가 1인·30일 자기상관
  시계열(iid 모집단이 아님)**이라 n=1500 KS가 과민하다는 점을 이해하고, **n-매칭 KS**를 추가했다.
  → *통계 도구를 돌리는 것과, 그 결과가 이 데이터에 타당한지 판단하는 것은 다르다.*

### 3.3 사람이 반드시 내려야 했던 결정

1. **원격 공유 DB에 180k행을 쓸 것인가**(되돌리기 어렵고 실데이터 60행과 공존) — 하네스가
   차단했고, 사람이 명시적으로 승인. AI가 임의로 진행하지 않았다.
2. **데이터 충실성 vs 서사 일관성의 우선순위** — 충돌 시 데이터 우선(사례 ①).
3. **튜닝을 멈출 지점(과적합 경계)** — "n=30 표본에 더 맞추면 과적합"이라 판단해 12/12 통과 시점에서
   추가 튜닝을 중단.
4. **삭제 금지 규칙 내에서의 우회** — 스모크 테스트 120행을 지우려다 DELETE가 차단되자, 규칙을
   뚫지 않고 `--skip-lines 120`으로 **이미 적재된 부분을 건너뛰어** 중복 없이 완성. *제약을 우회가
   아니라 설계로 흡수.*

## 4. 실패와 한계 (4-4)

### 4.1 가장 크게 헤맨 지점과 원인

| 헤맨 지점 | 원인 분류 | 무엇이 문제였나 |
|-----------|-----------|------------------|
| `ModuleNotFoundError: No module named 'app'` 반복 | 하네스/실행환경 | 스크립트를 파일로 실행하면 `sys.path[0]`가 스크립트 폴더라 프로젝트 루트의 `app`을 못 찾음. 루트를 path에 추가해 해결 |
| 작업 디렉터리 오염 | 하네스 | 앞선 `cd out/synthetic`가 셸에 남아 다음 명령이 엉뚱한 경로에서 실행됨. 절대경로 사용으로 회피 |
| `settings.DATABASE_URL` 로컬/원격 혼동 | 맥락 이해 부족 | 설정 병합 규칙을 모르면 조용히 틀림(4-3 사례 ②) |
| KS p-value 수치 버그 | AI 구현 결함 | 동점/상수 분포에서 급수 진동(4-3 사례 ③) |
| 서사 문장 조립 깨짐("새벽 새벽…", 문장 끊김) | 프롬프트/구현 | 문자열 단순 결합으로 중복·비문 발생. 템플릿을 다듬어 수정 |

### 4.2 한계

- **근본적 외삽**: 평시 클래스는 라벨 실데이터가 없어 응급의 비이벤트 기저에서 외삽했다. 본질적
  불확실성이 남는다.
- **1인 표본 일반화**: 각 클래스의 "진짜" 시그니처가 단 1명에서 왔다. 개인차가 클래스차로 오염됐을
  수 있다.
- **일부 지표의 CDF 한계**: night_aix처럼 1인·30일·계산오류가 섞인 지표는 모집단 생성기로 정확한
  분포 일치가 구조적으로 불가능하다. 평균·범위·분리 시그니처는 맞췄으나 KS 완전 일치는 못 한다.

### 4.3 다시 한다면

- **데이터부터 조회하고 시작**한다. night_aix를 추측으로 모델링한 뒤 되돌린 비용이 컸다 —
  "추측 금지, 경험분포 조회 먼저"를 첫 단계로.
- **공통 부트스트랩**(프로젝트 루트 path, DB URL 해석)을 한 곳에 모아 `app` import·로컬/원격
  실수를 구조적으로 차단한다.
- **검증 지표를 처음부터 두 종류**(전수 + n-매칭)로 설계해, 단일 KS의 과잉 기각에 휘둘리지 않는다.

## 5. 개인 성찰 (4-5) — **각 팀원이 직접 작성 (골격만 제공)**

> ⚠️ 지시서 6장: 개인 성찰은 **정의 복사 시 0점**, **팀원 간 복사 시 전원 감점**. 아래는 *채울 자리*일
> 뿐, 내용은 각자 자기 언어·자기 경험으로 써야 한다. 아래 "작업 흐름 연결 예시"는 참고용 뼈대이며
> 그대로 베끼면 안 된다.

각 팀원은 본인 이름으로 다음을 작성한다.

**① 다섯 개념에 대한 나의 깨달음** (개념별 — 한 줄 정의 / 처음 오해 vs 지금 이해 / 결정적 순간 /
한계·주의점). 다섯 개념(LLM · Prompt Engineering · Context Engineering · Harness Engineering ·
Agentic Coding)을 **따로 나열하지 말고 하나의 작업 흐름으로 연결**할 것.

> **작업 흐름 연결의 뼈대(참고용, 각자 재서술):** 이번 합성데이터 작업에서 다섯 개념은 이렇게
> 맞물렸다 — *`/goal` 로 출력 계약과 불변값을 못 박은 것(**Prompt**)*, *CLAUDE.md·plan·schema와
> id 1~60 실데이터를 골라 보여주고 그 외는 가린 것(**Context**)*, *그렇게 통제된 LLM이 서사·일과표를
> 생성하되 1440 배열은 못 만들게 한 것(**LLM의 한계 인정**)*, *권한 게이트·Stop 훅·검증 스크립트로
> 그 주위를 감싼 것(**Harness**)*, *단계별 산출물과 사람 승인 게이트로 진행한 것(**Agentic Coding**)*.
> 각자는 이 중 자신이 **직접 만졌던 순간**을 골라 자기 사건으로 풀어 쓸 것.

**② 나의 프로젝트 회고**: 내가 맡은 역할과 직접 내린 판단/결정, 가장 크게 배운 것 하나, AI를
쓰면서 바뀐 작업 습관·사고방식.

- 5.1 `[팀원 이름]` — `[작성]`
- 5.2 `[팀원 이름]` — `[작성]`
- …
