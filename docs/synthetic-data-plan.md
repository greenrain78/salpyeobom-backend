# 합성데이터 생성 계획 — ADL 3-클래스 학습용

> 고령자 ADL 모니터링 AI(평시/응급/사망 3-클래스 분류기) 학습용 합성데이터를
> "최대한 실제처럼" 생성하기 위한 설계 문서. 실데이터 `adl_raw_records` id 1~60
> (응급 1명 + 사망 1명, 각 30일)을 근거로 한다.

작성 맥락: 설계 논의 기록. 구현 착수 전 단계.

---

## 1. 목표

| 항목 | 값 |
|------|----|
| 산출물 | 평시·응급·사망 **각 1000명 × 60일** = 180,000행 |
| 용도 | 3-클래스 분류기 학습 (입력=하루치 지표, 출력=평시\|응급\|사망) |
| "실제처럼"의 기준 | 실데이터의 **분포 + 클래스 분리도(시그니처)** 재현 |
| 검증 | Streamlit 인터랙티브 대시보드 (실 vs 합성 비교) |

### 실데이터 한계 (출발점)
- "진짜 실데이터"는 **id 1~60 뿐** (응급 대상자 1명, 사망 대상자 1명, 각 30일 시계열)
- **평시(평소) 클래스는 검증된 실데이터 없음** → 응급/사망의 비이벤트 기저에서 파생 규칙 필요
- 즉 1명 × 2클래스 표본에서 1000명 × 3클래스를 일반화해야 함

## 3. 실데이터 분석 결과 (id 1~60)

### 3.1 단위·구조
- **1행 = 대상자 1명 × 하루**
- 60행 = (응급 1명 + 사망 1명) × 각 30일 시계열
- `created_at` 전부 2026-05-25 → 일괄 적재본

### 3.2 배열 컬럼 2종

| 종류 | 길이 | 컬럼 | 값 도메인 |
|------|------|------|-----------|
| 분단위 | 1440 (24h×60) | `place_code_1_list` | {0,1,10,20,30,254,255} (254/255 다수) |
| | | `aix_1_list` | 0이 71%, 비제로는 **33의 배수** (33,66,…,1000), 평균 354 |
| | | `sleep_depth_1_list` | {0,1,2,3,4}, 0이 75% |
| | | `outgoing_1_list` | {0,254,255}, 254·255가 99%+ |
| 시간단위 | 24 | `aix_h_list` | 0~759, 평균 110 |
| | | `temp_list` | 21.9~32.8 ℃ |
| | | `humi_list` | 42.3~71.7 % |
| | | `illu_list` | 0~125 lux |

### 3.3 클래스 시그니처 (응급 id1~30 vs 사망 id31~60)

가장 변별력이 큰 신호 = 합성 시 **반드시 보존**할 불변량.

| 지표 | 응급 | 사망 | 비고 |
|------|------|------|------|
| `bath_count_d` | 4~48 (평균 13) | **전부 0** | ★ 결정적 분리축 |
| `night_aix_ratio` | 0~20417 (평균 3584) | 0~15 (평균 6) | ★ 스케일 자체가 다름 |
| `aix_d` | 4~323 (평균 64) | 82~312 (평균 181) | |
| `total_aix_sum` | 56~75 (평균 63) | 0~278 (평균 179) | |
| `total_sleep_period` | 0~719 (평균 141) | 0~662 (평균 83) | |
| `outgoing_count_d` | 3~16 (평균 10) | 0~9 (평균 5) | |

- **시계열 추세**: 이벤트(응급/사망)일에 가까워질수록 지표가 추세적으로 변함 (단순 랜덤 아님). 사망 케이스는 D-day 직전 aix 급락·활동 소멸 패턴.

### 3.4 데이터 품질 함정 (스키마 문서 명시 — 합성 시 재현 여부 결정)
- `outgoing_1_list`의 **254/255 = 외출 아님, 센서 무신호 sentinel**
- `night_aix_ratio`·`sleep_start_time_d`·`sleep_end_time_d` = 원본 계산오류·결측 다수
- `sleep_start_time_d`는 스키마 문서엔 "HH:MM"이지만 **실제론 분(int) 문자열**("45","694")
- `care_recipient_id`는 과거 적재본에서 `"661.0"` 부동소수 표기 가능

> 상세 스키마는 [database-schema.md](database-schema.md)의 `adl_raw_records` 절 참조.

## 4. 생성 아키텍처 — 시나리오 기반

### 4.1 핵심 아이디어
LLM은 1440 배열을 토큰으로 직접 출력하지 않는다 (180k행 × 행당 ~5,856숫자 ≈ 10억 토큰,
비용·일관성·분포 모두 비현실적). 대신:

> **LLM = "환자 1명의 60일 인생을 쓰는 작가".**
> 의학적으로 그럴듯한 서사를 만들고, 그 서사에서 일별 정량 단서를 도출한다.
> 결정론적 Python 익스팬더가 그 단서로 1440 배열을 부풀린다.

### 4.2 파이프라인

```
goal 프롬프트 ─▶ LLM ─▶ [Layer1] 페르소나 + 60일 서사
                          "82세 독거. 무릎관절염 악화로 D-50부터 거동 점감.
                           D-7 감기로 수면 붕괴. D-day 새벽 욕실 낙상."
                          [Layer2] 서사 → 일별 정량 단서
                          (day별 활동레벨/수면창/목욕·외출 이벤트/aix 목표/환경기저)
                                     │
              결정론적 익스팬더(코드) ─▶ 1440 배열 + 스칼라
              + 불변 시그니처 강제 clamp + sentinel(254/255) + 33배수 양자화
                                     ▼
                          DB 적재 ─▶ Streamlit 검증 대시보드
```

### 4.3 시나리오 기반의 이점
- **인과 일관성**: 서사가 모든 지표를 한 방향으로 묶음 (관절염↑→거동↓→외출↓→야간이상↑)
- **다양성**: 1000명이 서로 다른 인생 스토리 → 자연스러운 분산
- **텍스트 필드**: `emergency_record`/`death_record`가 서사와 일치
- **평시 클래스**: 실데이터 없어도 LLM이 그럴듯한 정상 일상을 상상해 채움
- **검증·디버깅**: 각 행에 "이유(스토리)"가 붙어 추적 가능

### 4.4 ⚠️ 지켜야 할 규칙 — 시그니처는 코드가 강제
시나리오는 **클래스 경계가 아니라 "클래스 안에서의 변주"만** 흔든다. LLM이 창의적으로
쓰다 "사망인데 목욕 5회" 같은 걸 만들면 분류기 시그니처가 깨진다. 따라서 **익스팬더가
마지막에 불변 시그니처를 강제 clamp**:
- 사망 → `bath_count_d=0` 무조건
- `night_aix_ratio` → 클래스 봉투(envelope) 안으로 클램프
- `aix_1` → 33의 배수로 양자화, 71% 0
- `outgoing` 254/255 sentinel 비율 유지
- 서사는 봉투 *내부*의 디테일·추세·텍스트만 결정

## 5. goal 프롬프트 골격 (초안 v0)

```markdown
# 목표: 고령자 ADL 합성데이터를 "시나리오"로 생성

너는 고령자 1명의 60일을 쓰는 작가다. 의학적으로 그럴듯한 서사를 만들고,
그 서사에서 일별 행동 단서를 도출하라. (1440 원시배열은 코드가 전개한다.)

## 출력 단계
1. persona: age(70-95),sex,alone,vision,hearing,dosage,district,
   house_structure,room_no,bath_location, 기저활동량(low/mid/high)
2. scenario: 60일 서사 2~4문장. 클래스 패턴이 인과적으로 드러나게.
   - 평시: 안정된 일상, 이벤트 없음
   - 응급: D-day 응급사건으로 수렴하는 점진적 악화 서사
   - 사망: 말기 쇠약 → 활동·목욕 소멸 → D-day 사망 서사
3. daily[60]: 서사를 day별 정량 단서로
   (활동레벨/수면창/목욕횟수/외출/aix추세/환경기저)
4. records: 서사와 일치하는 emergency_record 또는 death_record 텍스트

## 불변 시그니처(서사가 위반 못 함 — 코드가 clamp하지만 단서도 맞춰라)
- 사망=목욕0, night_aix 스케일 클래스별 상이, aix 33배수,
  outgoing 254/255=sentinel, sleep_depth {0~4}
## 다양성 요구
1000명 서로 다른 질환·생활·악화속도. 같은 스토리 반복 금지.
```

## 6. 단계별 계획 (각 단계 게이트)

| Phase | 내용 | 산출물 |
|-------|------|--------|
| **0** | 실데이터 프로파일 고정 (검증 기준선) — id 1~60에서 클래스별 분포·전이행렬·이벤트수렴 추세 추출 | `profiles/real_baseline.json` |
| **1** | goal 프롬프트 확정 (§5 다듬기) | `prompts/goal_generation.md` |
| **2** | 생성 파이프라인 — LLM 호출(서사·단서) + 익스팬더(→1440배열) + DB 적재 | `gen/` 모듈 + 180k행 |
| **3** | 검증 대시보드 — 실 vs 합성 분포 겹쳐보기·KS test·클래스 분리도(2D 투영)·대상자 시계열 드릴다운 | `dashboard/app.py` (Streamlit) |

### 합격 기준 (Phase 3)
- 주요 지표 KS test p>0.05 **또는**
- 클래스 분리도(합성 학습→실데이터 검증 분류 성능)가 실데이터 기준선과 ±x% 이내
