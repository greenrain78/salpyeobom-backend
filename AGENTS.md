# AGENTS.md — AI 에이전트 작업 가이드

> 이 파일은 AI 에이전트가 이 프로젝트에서 어떻게 작업을 라우팅하고, 어디까지 접근할 수 있으며,
> 어떤 검증 루프를 거쳐야 하는지를 정의합니다.

---

## 라우터 — 의도 분석 및 작업 분류

요청을 받으면 곧바로 실행하지 말고, 먼저 아래 분류표에 따라 의도를 파악합니다.

| 요청 유형 | 판단 기준 | 처리 방식 |
|-----------|----------|-----------|
| **모호한 요청** | 범위, 위치, 영향 범위 불명확 | 사용자에게 되묻기 (실행 금지) |
| **명확한 코드 작업** | 파일, 엔드포인트, 변경 범위 명확 | 하네스 루프 진입 |
| **조회/분석** | 코드 읽기, 설명, 검색 | 즉시 수행 (변경 없음) |
| **일반 질문** | 개념 설명, 방법론 | 일반 대화 |

### 되물어야 하는 상황 (실행 전 확인 필수)

- "기능 추가해줘" → **어떤 기능인지, 어떤 엔드포인트인지** 묻기
- "DB 수정해줘" → **어떤 모델의 어떤 필드인지** 묻기
- "성능 개선해줘" → **어떤 부분이 느린지, 측정값이 있는지** 묻기
- "리팩토링해줘" → **어떤 파일/함수를, 왜, 어떤 방향으로인지** 묻기

## 컨텍스트 매니저 — 필요한 정보만 로드

작업 시 전체 코드베이스를 읽지 말고, **현재 작업과 직접 관련된 파일만** 로드합니다.

### 항상 고정(pin)되는 컨텍스트

- `CLAUDE.md` — 절대 규칙
- `app/config.py` — 환경 설정
- `app/database.py` — DB 연결 설정
- `app/core/exceptions.py` — 에러 패턴

### 작업별 필요 파일

| 작업 | 로드할 파일 |
|------|------------|
| 새 엔드포인트 추가 | 관련 `routers/*.py` + `schemas/*.py` + `models/*.py` |
| 인증 수정 | `core/security.py`, `core/dependencies.py`, `routers/auth.py` |
| 모델 변경 | `docs/database-schema.md` (먼저 읽기 — 전체 스키마 파악) + `models/*.py` + 관련 `schemas/*.py` + 기존 마이그레이션 — 절차는 [CLAUDE.md "DB 스키마 변경 패턴"](./CLAUDE.md#db-스키마-변경-패턴) |
| DB 구조 조회/질문 | `docs/database-schema.md` (8개 테이블 + ERD + 필드 의미가 한 문서에 정리됨) |
| 테스트 작성 | `tests/conftest.py` + 대상 `routers/*.py` |
| 배포 스크립트 | `scripts/` + `Makefile` |

## 하네스 루프 — 코드 작업 실행 절차

코드를 변경할 때는 반드시 이 루프를 따릅니다:

```
1. 계획 (Plan)
   → 변경할 파일 목록 확인
   → 영향 범위 파악 (다른 파일에 영향 있는지)
   → 마이그레이션 필요 여부 확인

2. 실행 (Execute)
   → 코드 작성 (CLAUDE.md 규칙 준수)
   → 타입 어노테이션 포함

3. 검증 (Verify)
   → make fix     # 포맷/린트 자동 수정
   → make check   # 전체 품질 검사
   → 테스트 실패 시 → 수정 루프

4. 완료 (Done)
   → make check 통과 확인
   → 변경 내용 요약
```

**MAX_ATTEMPTS = 3**: 검증이 3회 연속 실패하면 실행을 멈추고 사용자에게 보고합니다.

## 도구 경계 — 접근 권한 정의

### 허용된 도구

| 도구 | 범위 | 용도 |
|------|------|------|
| 파일 읽기 | `app/`, `tests/`, `scripts/`, 설정 파일 | 코드 이해 |
| 파일 쓰기 | `app/`, `tests/`, 설정 파일 | 코드 작성 |
| `uv run *` | 프로젝트 내 | 앱 실행, 테스트, 린터 |
| `make *` | Makefile 타겟 | 개발 명령어 |
| `git status`, `git diff`, `git log` | 읽기 전용 | 변경 내용 확인 |
| `git add`, `git commit` | 명시적 요청 시만 | 커밋 (자동 실행 금지) |

### 금지된 도구

→ 코드/데이터 안전 관련 절대 금지 항목은 [CLAUDE.md "절대 규칙"](./CLAUDE.md#절대-규칙-위반-불가) 참고.
추가로 AI 작업 시 다음도 금지: `rm -rf`, `sudo rm`(시스템 파일 삭제), 의도치 않은 외부 API 직접 호출.

## 워커 역할 분리

복잡한 작업에서 AI는 두 역할을 **분리**하여 수행합니다:

### 코드 작성 워커 (WRITE_CODE)
- 순수하게 코드만 작성
- 설명 최소화, 기능적 코드에 집중
- 타입 어노테이션 포함

### 검증 워커 (VERIFY)
- 작성된 코드의 논리적 오류 탐지
- 보안 취약점 확인 (SQL injection, 인증 누락 등)
- 테스트 케이스 누락 확인

> 같은 AI가 쓰고 검토하면 실수를 놓치기 쉽습니다. 역할을 의식적으로 분리하세요.

## 피드백 루프 — 에러 처리 방식

### `make check` 실패 시

```bash
# 1. ruff 오류 → 자동 수정 시도
make fix

# 2. mypy 오류 → 타입 어노테이션 추가/수정
# 오류 메시지 읽고 해당 파일의 타입 어노테이션 수정

# 3. pytest 오류 → 테스트 분석 후 코드 또는 테스트 수정
# 테스트 자체의 문제인지, 구현의 문제인지 판단

# 4. 커버리지 미달 → 테스트 추가
uv run pytest --cov=app --cov-report=term-missing
# 빨간색으로 표시된 uncovered 라인 확인 후 테스트 추가
```

### pre-commit hook 실패 시

커밋이 거부되면 자동 수정을 시도하고 다시 스테이징:
```bash
make fix
git add -u
# 다시 커밋 시도
```
